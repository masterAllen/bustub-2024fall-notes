# 为什么这里还要再 `ScanKey(new_key, ...)`

你可能会问：  
前面不是已经做过一次主键冲突检查了吗，为什么这里又查一遍？

原因是这两次检查作用不同：

## 前面的检查
是逻辑语义检查：
- “这个新主键值会不会和别的当前可见 tuple 冲突？”

## 这里的检查
是索引维护检查：
- “这个新 key 到当前 RID 的映射是不是已经存在？”
- 如果已经有了，就别重复插一遍

所以这里不是重复做唯一性判断，而是在避免：
- 重复插相同 `(key, rid)` entry

它会先走主键索引找 `a = 1`：

- 结果什么也找不到

但这其实是错的，因为按它的快照时间，它本来应该还能看到旧版本 `(a=1, b=100)`。

---

## 不删 old_key 之后会怎样

如果你保留：

```text
1 -> r7
2 -> r7
```

那么 `T_old` 查 `a = 1` 时：

1. 索引命中 `r7`
2. 拿到 base tuple（现在可能是 `a=2`）
3. 再沿 undo log 回退
4. 恢复出它在旧时间点应该看到的版本 `(a=1, b=100)`

这样 MVCC index scan 才成立。

---

## 所以这里的索引不是“当前状态索引”，而是“历史入口”

这就是最关键的观念转换。

在 Task 4.2 以后，主键索引的职责不再只是：

**“告诉我当前这个 key 对应哪条记录”**

而更像是：

**“给我一个入口，让我能找到那个 RID，然后再由 MVCC 逻辑决定当前事务到底能看到这个 RID 的哪个版本”**

因此旧 key 不能轻易删。

---

## 为什么这只对主键索引这么做

因为这一阶段题目只要求你重点处理 primary key index 的 MVCC 行为。  
它把主键索引当成“稳定入口”。

而普通 secondary index 在这个任务里没有要求做到同等历史语义，所以你现在看到非主键索引仍然是删旧插新。

---

## 代价是什么

代价就是你刚才感觉到的“怪”：

- 一个 RID 可能被多个旧 key 指到
- 索引里会保留历史映射
- 所以后面逻辑必须额外判断：
  - 命中的是不是自己
  - 命中的 RID 对当前事务是否真的可见

这也是为什么 Task 4.2 开始代码复杂度突然上去了。

---

## 一句话总结

`old_key` 不删除，是因为：

**BusTub 需要让旧事务仍然能通过旧主键值找到同一个 RID，再通过 undo log 回退到历史版本。**

如果删掉 `old_key -> RID`，MVCC 下的 index scan 就会丢失历史可见性。
